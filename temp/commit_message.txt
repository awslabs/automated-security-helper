feat(mcp): Add comprehensive filtering and improve MCP server stability

This commit introduces major improvements to the ASH MCP server including
adaptive polling, robust error handling, flexible filtering capabilities,
and comprehensive documentation.

## Major Changes

### 1. Adaptive Polling for Long-Running Scans
- Optimized polling intervals for 2-5 minute scan duration
- 5s (initialization) → 8s (early) → 10s (main) → 15s (heartbeat)
- Reduces network traffic by 80-93% compared to 1s polling
- Prevents client disconnections during long scans

### 2. Comprehensive Error Handling
- All MCP communication wrapped in try-except blocks
- Graceful handling of ClosedResourceError and CancelledError
- Server continues monitoring even if client disconnects
- Fixed error checking logic to properly detect actual errors
- Prevents server crashes with detailed error logging

### 3. Flexible Response Size Filtering (filter_level)
- "minimal" (~1-2KB): Basic status for fast polling/health checks
- "summary" (~5-15KB): Metadata and counts for dashboards/CI/CD
- "full" (default): Complete results for detailed analysis
- Maintains full backward compatibility

### 4. Content Filtering by Scanner and Severity
- scanners: Filter by specific tools (e.g., "bandit,semgrep")
- severities: Filter by severity levels (e.g., "critical,high")
- Filters can be combined for maximum flexibility
- Includes _content_filters metadata showing applied filters

### 5. New Lightweight Tools
- get_scan_summary: Dedicated lightweight summary (~5-10KB)
- get_scan_result_paths: File paths for all result formats (~2-5KB)
- Enables progressive data fetching and file-based access

### 6. Bug Fixes
- Fixed Pydantic model initialization in asharp_model.py
- Renamed 'filter' parameter to 'filter_level' to avoid Python builtin conflict
- Fixed error handling logic in get_scan_results and get_scan_summary
- Corrected success key checking to prevent premature returns
